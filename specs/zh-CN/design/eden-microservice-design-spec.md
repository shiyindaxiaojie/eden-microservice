# Eden Microservice 设计规范

## 1. 定位

`eden-microservice` 是轻量级微服务控制面。它以单进程优先的部署形态提供注册中心、配置中心、API
网关、控制台、认证授权、告警通知和集群协同能力，用来降低小中型系统对 Nacos、Consul、
ZooKeeper 和 Spring Cloud Gateway 的组合依赖。

它不是完整 Service Mesh、通用对象存储、完整 KMS、业务配置 schema 平台或外部数据库替代品。

## 2. 一期目标

一期在已有注册中心能力上新增两个一级领域：

| 领域 | 目标 | 关键接口 |
| --- | --- | --- |
| 配置中心 | 支持配置发布、查询、删除、历史、md5 版本、长轮询变更通知和 Nacos Config HTTP 兼容 | `/v1/config*`、`/nacos/v1/cs/*` |
| API 网关 | 支持路由管理、健康实例选择、HTTP 反向代理、超时、基础过滤器和注册中心服务发现 | `/v1/gateway/*`、独立网关监听端口 |

注册中心、命名空间、RBAC、设置、通知、告警和集群管理保持现有方向，并逐步从
`eden-registry` 命名迁移到 `eden-microservice`。

## 3. 领域边界

| 领域 | 拥有 | 不拥有 |
| --- | --- | --- |
| Naming | 服务、实例、健康、心跳、服务发现、拓扑 | 配置文件内容、网关转发规则 |
| Config | 配置身份、内容、md5、发布、查询、历史、监听、兼容 API | 服务实例生命周期、网关流量转发 |
| Gateway | 路由、匹配规则、上游选择、过滤器、转发、网关指标 | 配置文件业务语义、服务注册写入 |
| Console | 管理视图、表单、权限可见性、运维操作入口 | 重新定义领域语义 |
| Cluster | AP 复制、CP 共识、节点成员和内部同步 | 业务资源的领域校验 |

## 4. 架构原则

### 4.1 单进程优先

默认形态仍是一个 server 进程。配置中心和网关由同一控制面装配，但网关数据面可以使用独立监听端口，避免控制台和代理流量共享路径空间。

### 4.2 兼容优先但不牺牲原生模型

Nacos Config、Nacos Naming、Consul Registry 的兼容 API 用于迁移存量应用。新功能和控制台优先使用清晰的原生 `/v1/*` API。兼容接口只做适配，不反向污染内部领域模型。

### 4.3 管理面与运行面分离

管理面用于控制台、管理员、开发者和自动化脚本；运行面用于业务进程读取配置、注册发现或经过网关访问上游。高频运行路径不得依赖复杂控制台表单逻辑。

### 4.4 AP 与 CP 都要有清晰写入路径

Standalone 写入本地持久化。AP 集群允许写入后向 peers 扩散并最终一致。CP 集群中的关键持久资源写入必须通过 Leader 和 Raft 提交。

## 5. 目标模块

```text
internal
├─ catalog              # 已有注册中心领域
├─ configcenter         # 新增配置中心领域
├─ gateway              # 新增网关路由、匹配、转发领域
├─ adapter
│  ├─ nacos             # Nacos Naming + Config 兼容
│  └─ consul            # Consul Registry 兼容
├─ transport
│  ├─ http              # 控制台和原生 HTTP API
│  ├─ rpc               # gRPC API
│  └─ gateway           # 可选的网关数据面监听装配
└─ cluster              # AP/CP 复制和共识
```

## 6. 控制台导航

控制台应把配置和网关作为一级菜单：

```text
概览 -> 服务列表 -> 配置管理 -> 路由管理 -> 命名空间 -> 节点管理 -> 访问控制 -> 系统设置
```

`配置管理` 用于 Config 资源 CRUD、历史和发布。`路由管理` 用于 Gateway 路由、启停、匹配条件和上游策略。二者都不应放到 `服务列表` 的子页里。
